# สรุปสถาปัตยกรรมการทำงานของ WebAssembly (WASM) ในระบบ Strategy

เอกสารนี้สรุปการทำงานของโมดูล WebAssembly (WASM) ที่ใช้สำหรับการประมวลผลกลยุทธ์การเทรด (Trading Strategy) โดยเน้นไปที่การแบ่งแยกหน้าที่ระหว่าง CPU และ GPU เพื่อให้เกิดประสิทธิภาพสูงสุดในการประมวลผลอินดิเคเตอร์

## 1. ฝั่ง JavaScript (Web API)
สคริปต์ฝั่ง Frontend ที่ทำหน้าที่เรียกใช้งานโมดูล WASM ถูกแบ่งออกเป็น 2 ชั้นการทำงาน:

- **`strategy_cpu.js` (CPU Mode)**: 
  - ทำหน้าที่เรียกใช้งาน WASM สำหรับประมวลผลการคำนวณ **แบบเรียลไทม์ (ทีละ Tick)** ผ่าน CPU
  - เหมาะกับการรับข้อมูลราคาแบบสดๆ จาก WebSocket (เช่น Deriv API) และนำไปคำนวณกลยุทธ์ทันที
  - ไม่จำเป็นต้องใช้การ์ดจอ สามารถรันข้ามแพลตฟอร์มหรือมือถือได้เสถียรกว่า

- **`strategy_gpu.js` (GPU Mode)**:
  - ทำหน้าที่เป็น Wrapper เชื่อมต่อกับเทคโนโลยี **WebGPU** เพื่อสั่งให้การ์ดจอทำงานแบบขนาน (Parallel Computing)
  - เหมาะสำหรับ **การทำ Backtest ฐานข้อมูลขนาดใหญ่** (ข้อมูลราคาเป็นแสนๆ แท่งเทียน) ซึ่งจะลดระยะเวลาในการคำนวณได้อย่างมหาศาล
  - มีระบบ Fallback ถ้าระบบพบว่าเบราว์เซอร์หรือเครื่องนั้นๆ ไม่รองรับ WebGPU ก็จะสลับไปใช้งาน CPU ให้โดยอัตโนมัติ (แต่ไม่เหมาะกับข้อมูลแบบ Real-time ทีละ Tick เนื่องจาก Overhead สูงเกินจำเป็น)

## 2. ไฟล์ `.wasm` ที่ถูกสร้างออกมา (Compiled Binary)
เมื่อทำการคอมไพล์โค้ดจากภาษา Rust ออกมา จะได้ไฟล์ WebAssembly เพื่อใช้งานบนเว็บเบราว์เซอร์ **เพียง 1 ไฟล์เท่านั้น**:

- `RustLib/indicator_math/wasm_dist/indicatorMath_ULTRA_Rust_bg.wasm`
  - ไฟล์นี้คือไฟล์ Binary เดี่ยวๆ ที่บรรจุตรรกะทั้งหมด ตั้งแต่สูตรคำนวณอินดิเคเตอร์ฝั่ง CPU ไปจนถึงทิศทางการแจกแจงเมมโมรี่ฝั่ง GPU ซึ่งรวมอยู่ในไฟล์เดียวเพื่อการโหลดขึ้นใช้งานที่ง่ายและรวดเร็ว

## 3. ไฟล์ต้นฉบับฝั่ง Rust (Rust Source Files)
ถ้าเจาะลึกไปที่โค้ดต้นฉบับภายในโฟลเดอร์ฝั่ง Rust (`RustLib/indicator_math/src/wasm/`) จะพบว่าการทำงานถูกแบ่งเป็น 2 ระบบชัดเจน:

1. **`cpu.rs`**
   - **สิ่งที่จำลองให้ JS เรียกใช้:** คลาส `WasmAnalysisGenerator`
   - **หน้าที่หลัก:** รอต้อนรับราคาหลุดข้อมูลแบบทีละ Tick ของ `append_tick()` รับข้อมูลปุ๊บก็โยนเข้าเครื่องคิดเลข Rust แล้วส่งกลับไปยังหน้าจอผู้ใช้

2. **`gpu.rs`**
   - **สิ่งที่จำลองให้ JS เรียกใช้:** คลาส `GpuAnalysisManager`
   - **หน้าที่หลัก:** เตรียมข้อมูลดิบแบบชุดใหญ่ (Batch processing) ปรับขนาดและจองท่อส่งความจำ (Buffer) ไปยัง VRAM ของการ์ดจอโดยรันผ่านไลบรารีเบื้องหลังที่ชื่อ `wgpu` ช่วยให้อุปกรณ์ทำงานพร้อมๆ กันได้

3. **`sma_shader.wgsl`**
   - **หน้าที่หลัก:** เป็นโค้ดภาษา Shader พื้นฐาน (คล้ายๆ C/C++) ซึ่งจะถูกโหลดและอัดเข้าสู่ชิปของการ์ดจอโดยตรง ทำหน้าที่เป็น "สมองดั้งเดิม" ที่รับคำสั่งการคำนวณ SMA ควบคู่ไปกับเส้นที่จองคิวไว้ใน VRAM อย่างขนานกัน

---

**บทสรุปโดยสังเขป**
ทั้งระบบทำงานเป็นหนึ่งเดียวกันโดยมี **JavaScript เป็นผู้ควบคุมจังหวะ**, มี **Rust WASM เป็นสมองประมวลผลที่รวดเร็ว**, และดึงดูดพลังของ **WebGPU เป็นตัวกระตุ้นให้การประมวลผล Backtest เสร็จเร็วพริบตา**
